What is Column Masking?

Column masking allows you to hide or obfuscate sensitive data in specific columns while still allowing users to query the table. Instead of seeing the actual values, users see masked values (like xxx, ****, or hashed values) based on the masking policy you define.

💡
Use Case Example

A customer table contains email addresses and credit card numbers. With column masking, you can ensure that diuser sees xxx instead of actual email addresses, while administrators see the real data. This protects sensitive information while maintaining data accessibility.

✅

IBM watsonx.data now supports Apache Ranger policies to allow comprehensive data security on integrating with multiple governance tools and engines. Column masking is a key component of this security framework.

Procedure

Creating a Column Masking Policy

Follow these steps to create a column masking policy in Apache Ranger that protects sensitive data in specific columns.

Step 1: Access Apache Ranger Service Manager

1
Log in to Apache Ranger

Access the Apache Ranger console using your admin username and password.

2
Navigate to Service Manager

From the Ranger home page, go to Service Manager and locate your PRESTO service (e.g., watsonx-presto).

3
Open the service policies page

Click on the service name to open the policies page where you can add masking policies.

Apache Ranger → Service Manager → watsonx-presto
🛡️
Screenshot: Opening Service Policies Page
Service Manager → Click on watsonx-presto service name
PRESTO Service

Step 2: Add Masking Policy

4
Click "Add New Policy" for Masking

On the policies page, look for the Masking tab or section, then click Add New Policy to create a new masking policy.

5
Fill in Policy Details

In the Policy Details section, provide a descriptive policy name (e.g., mask-email-diuser) and an optional description explaining the policy's purpose.

Step 3: Define Resources

6
Select catalog, schema, table, and column

In the Resources section, specify the exact catalog, schema, table, and column for which this masking policy applies. For example: Catalog = iceberg_data, Schema = dischema, Table = customer, Column = email.

🔑
Column Selection is Critical

Unlike access policies, masking policies require you to specify the column name that contains sensitive data. The column name must match exactly (case-sensitive) with the actual column in the table.

Step 4: Configure Masking Conditions

7
Add user and permissions

In the Masking Conditions section, select the user (e.g., diuser) and set the Permissions to Select.

8
Set Select Masking Option to Custom

In the Select Masking Option dropdown, choose Custom to define your own masking value.

9
Enter the masked value

In the masking value field, enter the value that will replace the actual data. For example, enter xxx to mask all values in the column with "xxx" for diuser.

ℹ️
Masking Options Available

Apache Ranger provides several built-in masking options: Redact (replace with X), Partial mask (show first/last N characters), Hash (SHA256), Nullify (replace with NULL), Date: show only year, and Custom (your own value). Choose the option that best fits your security requirements.

Key Points

Important Considerations

  • ✓
    Column-level granularity — Masking policies apply to specific columns, not entire tables. You can mask multiple columns with different policies.
  • ✓
    Transparent to users — Users don't need to modify their queries. The masking is applied automatically by Ranger.
  • ✓
    Different masks for different users — You can create different masking policies for different users on the same column (e.g., partial mask for some, full redaction for others).
  • ✓
    Column names must be exact — The column name in your policy must match the actual column name in the table (case-sensitive).
  • ✓
    Policy cache refresh — Changes to masking policies may take a few seconds to propagate based on your policy cache time setting.
  • ✓
    Combine with access policies — Column masking works alongside access control policies. Users still need SELECT permission on the table.
  • ✓
    Performance impact — Masking is applied at query time, which may have a slight performance impact on large result sets.
  • ✓
    Audit trail — All masked data access is logged in Ranger audit logs for compliance and security monitoring.
