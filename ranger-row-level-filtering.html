What is Row-Level Filtering?

Row-level filtering allows users to access a subset of rows in a table based on specific conditions. Instead of granting or denying access to an entire table, you can define policies that filter which rows a user can see when they query the table.

💡
Use Case Example

A sales table contains data for all regions. With row-level filtering, you can ensure that diuser only sees rows where region='US-WEST', while User2 only sees rows where region='EU-CENTRAL' — all from the same table.

Prerequisites

Before You Begin

Ensure you have the following in place before creating row-level filtering policies:

  • ✓
    IBM watsonx.data instance — A running watsonx.data deployment (SaaS or On-Premises)
  • ✓
    Apache Ranger provisioned — Apache Ranger server is installed and accessible
  • ✓
    Ranger integration enabled — Apache Ranger policy is enabled in watsonx.data. See Enabling Apache Ranger policy for resources
  • ✓
    User created — The user (e.g., diuser) exists in both watsonx.data and Apache Ranger with matching usernames
  • ✓
    Table exists — The target table is registered in a catalog accessible by watsonx.data
01
Procedure

Creating a Row-Level Filter Policy

Follow these steps to create a row-level filtering policy in Apache Ranger that restricts which rows a user can access.

Step 1: Access Apache Ranger Service Manager

1
Log in to Apache Ranger

Access the Apache Ranger console using your admin username and password.

2
Navigate to Service Manager

From the Ranger home page, go to Service Manager and locate your PRESTO service (e.g., watsonx-presto).

3
Open the service policies page

Click on the service name to open the policies page where you can add row-level filter policies.

Apache Ranger → Service Manager → watsonx-presto
🛡️
Screenshot: Opening Service Policies Page
Service Manager → Click on watsonx-presto service name
PRESTO Service

Step 2: Add Row-Level Filter Policy

4
Click "Add New Policy"

On the policies page, click the Add New Policy button to create a new row-level filter policy.

5
Fill in Policy Details

In the Policy Details section, provide a descriptive policy name (e.g., row-filter-diuser-customer) and an optional description explaining the policy's purpose.

Step 3: Define Resources

6
Select catalog, schema, and table

In the Resources section, specify the exact catalog, schema, and table for which this row-level filter applies. For example: Catalog = iceberg_data, Schema = dischema, Table = customer.

Step 4: Configure Row Filter Conditions

7
Add user and permissions

In the Row Filter Conditions section, select the user (e.g., diuser) and set the Permissions to Select.

8
Enter the Row Level Filter expression

In the Row Level Filter field, enter the filter condition. For example: c1=1 (only rows where column c1 equals 1 will be visible to diuser).

💡
Filter Expression Examples

Row-level filter expressions use SQL-like syntax. Examples: region='US-WEST', department='Sales', year>=2023, status='active' AND region='EU'

Apache Ranger → Add New Policy → Row Filter Conditions

Step 5: Save the Policy

9
Click Save

Scroll to the bottom of the policy form and click Save. The row-level filter policy is now active in Apache Ranger.

Apache Ranger → Add New Policy → Save

02
Verification

Verifying Row-Level Access Control

Now test the row-level filter policy by logging in as diuser and querying the table. The results should only include rows that match the filter condition.

1
Log in to watsonx.data as diuser

Access the watsonx.data console using diuser's credentials.

2
Open Query Workspace

From the navigation menu, click Query workspace and select the Presto engine.

3
Run a query to access the table

Execute a simple SELECT query against the table:

SQL — watsonx.data Query Workspace (Presto) 
SELECT * FROM iceberg_data.dischema.customer;

Row-Level Filtering is Working

diuser can now only see rows where column c1 equals 1. All other rows in the table are filtered out automatically by Apache Ranger. The user doesn't need to add a WHERE clause — the filter is enforced transparently at the policy level.

Key Points

Important Considerations

  • ✓
    Filter expressions use SQL syntax — You can use standard SQL conditions like =, !=, >, <, AND, OR, IN, etc.
  • ✓
    Filters are transparent to users — Users don't need to modify their queries. The filter is applied automatically by Ranger.
  • ✓
    Multiple users, different filters — You can create different row-level filter policies for different users on the same table.
  • ✓
    Column names must be exact — The column names in your filter expression must match the actual column names in the table (case-sensitive).
  • ✓
    Policy cache refresh — Changes to row-level filter policies may take a few seconds to propagate based on your policy cache time setting.
  • ✓
    Combine with column masking — Row-level filtering can be used alongside column masking policies for comprehensive data protection.
